任务

Task 1：实现final_suspend的调度逻辑

协程task结束后的调度逻辑通过promise_type的 final_suspend() 函数实现，如果简单地让 final_suspend() 返回 std::always_suspend ，当子协程任务结束后，执行权会继续沿着调用栈向上转移直到找到非suspend状态的协程，于是会发生下图过程

  sequenceDiagram
    participant Caller as 协程调用者
    participant Task1 as task1
    participant Task2 as task2

    Caller-->>Task1: 执行协程
    Task1-->>Task2: co_await task2
    Task2-->>Task1: initial_suspend(return std::suspend_always) 
    Task1-->>Task2: await_suspend(return task2.handle())
    Note over Task1: 陷入suspend状态
    Note over Task2: return
    Task2-->>Caller: final_suspend(return std::suspend_always)
    Note over Task1: return 部分被跳过

为了实现正确的嵌套调用逻辑，task应当在被嵌套调用时记录父级task的 handle到状态变量continuation以便任务结束后正确转移执行权到父级task，在co_return后调用final_suspend返回一个自定义awaiter转移执行权到continuation，正确实现逻辑如下

  sequenceDiagram
    participant T1 as Task1 (Coroutine)
    participant A as Awaiter (co_await)
    participant B as Awaiter (co_return)
    participant T2 as Task2 (Coroutine)
    participant S as Runtime/Executor

    Note over T1: 正在执行业务逻辑...
    T1->>A: 1. 执行 co_await Task2()
  
    Note over A: Awaiter 协议开始
    A->>A: 2. 调用 await_ready()
    Note right of A: 返回 false (表示未就绪)
  
    T1->>T1: 3. Task1 暂停并保存状态 (挂起)
  
    A->>A: 4. 调用 await_suspend(T1_handle)
    Note right of A: 将 T1 句柄存入 Task2 的 Promise
    A-->>T2: 5. 返回 Task2_handle (切换执行权)

    Note over T2: Task2 开始运行...
    T2->>B: 6. 执行 co_return
  
    B->>B: 7. 调用await_ready()
    Note right of B: 返回 false (表示未就绪)
    B->>B: 8. 调用await_suspend(continuation)
    Note right of B: 转移执行权到continuation
    T2->>T1: 9. 从 Promise 找到 T1_handle 并 Resume
  
  
    Note over T1: Task1 唤醒
  
    T1->>A: 10. 调用 await_resume()
    Note over A: 生命周期结束
    A-->>T1: 返回数据/结果
    Note over T1: 继续执行后续代码

promise_base保存着协程的状态信息，为其新增成员变量以及修改该成员变量的方法

在task的awaiter里修改 awaiter_suspend()逻辑保存task1协程句柄到task2的promise变量里

再新增 final_awaiter,它只在final_suspend中调用,因此可以在promise_base作用域内声明，注意将其声明为友元类以获取protected变量访问权；随后修改final_suspend调度逻辑

若continuation指向nullptr则返回 std::noop_coroutine(),沿着调用栈向上寻找未挂起的调用者。

Task 2: 为task添加 detach 状态

为了保证协程资源的即时释放,tinyCoro的task使用RAII的方式在析构时销毁协程句柄,但如果协程正在执行中则会导致未定义行为。此时执行引擎不会感知到task的存在，将task以右值语义提交给engine后，应当将task的协程句柄的所有权也转交给engine，同时调用 task.detach()方法将task置为 detach 状态，此后协程资源释放的义务由task转移到执行引擎，执行引擎执行完毕后调用 clean()释放协程资源。

新增一个状态枚举类，并为promise_base新增成员变量记录当前状态，初始化为 normal；同时新增获取、修改、判断是否detach的接口。

接下来添加 task.detach()接口，对其关联的promise设置detach状态，并移除自身协程句柄

最后增加工具函数clean()，由执行引擎调用清理协程句柄对应资源。
