Task 1：完善 engine 初始化以及异步 I/O 支持

在异步 IO 场景下，engine 需要同时管理两类任务：普通任务（由 AtomicQueue 存储）和 IO 任务（由 io_uring 管理）。为此，tinyCoro 为 engine 添加了两个计数器：

这两个变量体现了 IO 操作的两个阶段：

  • wait_submit：协程已经创建了 IO 操作（填充了 sqe），但尚未调用 io_uring_submit
  • running：IO 已提交给内核，正在异步执行中

协程发起 IO 的过程发生在工作线程内部（不跨线程），因此这两个计数器不需要原子变量保护，使用普通 size_t 即可。

poll_submit：核心调度循环

poll_submit 是 engine 的核心函数，负责 IO 的提交、等待、处理三个阶段：

流程分析：

  1. 提交阶段：将所有 wait_submit 的 IO 批量提交，更新 running 计数器
  2. 等待阶段：通过 eventfd 阻塞，直到有 IO 完成（cqe 产生）或被 wake_up 唤醒
  3. 取出阶段：批量读取完成的 CQE 到缓冲区
  4. 处理阶段：遍历每个 CQE 触发对应的协程继续执行，并更新 running 计数器

eventfd 唤醒机制

工作线程在无任务时阻塞在 eventfd 读操作上，避免 CPU 空转。问题在于：当 scheduler 向一个正在阻塞的工作线程派发新任务时，如何唤醒它？

解决方案是通过 eventfd 写值唤醒：

当工作线程阻塞在 wait_eventfd 时，任何对 eventfd 的写操作都会立即唤醒它。

唤醒事件的三种来源

工作线程被唤醒可能有三种情况：

  • case1： 有新任务提交（wake_by_task）
  • case2： 有新 IO 提交（已废弃，跨线程 IO 已移除）
  • case3： 有 IO 已完成（wake_by_cqe）
